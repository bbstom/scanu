# 后台配置管理说明

## 功能概述

管理员可以通过后台管理界面动态调整系统配置，无需重启服务即可生效。

## 访问后台

### 1. 首次设置

访问 `/admin/setup` 创建管理员账号：

```
URL: http://localhost:3000/admin/setup
```

填写：
- 用户名
- 密码
- 确认密码

### 2. 登录

访问 `/admin` 登录：

```
URL: http://localhost:3000/admin
```

### 3. 配置管理

登录后访问 `/admin/settings`：

```
URL: http://localhost:3000/admin/settings
```

## 可配置项

### API 配置

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| ETHERSCAN_API_KEY | Etherscan API Key | 空 |
| TRONGRID_API_KEY | TronGrid API Key | 空 |
| COINGECKO_API_KEY | CoinGecko API Key | 空 |

**说明**:
- 如果不配置，系统会使用免费公共 API
- 配置后可以提高请求限制和稳定性
- 环境变量优先级高于数据库配置

### 缓存配置

| 配置项 | 说明 | 默认值 | 推荐范围 |
|--------|------|--------|----------|
| CACHE_BALANCE_TTL | 余额缓存时间（秒） | 60 | 10-3600 |
| CACHE_TRANSACTIONS_TTL | 交易缓存时间（秒） | 120 | 10-3600 |

**调整建议**:
- **实时性要求高**: 减少 TTL（30-60 秒）
- **API 调用受限**: 增加 TTL（300-600 秒）
- **开发测试**: 设置较短 TTL（10-30 秒）
- **生产环境**: 使用默认值或稍高（60-300 秒）

**影响**:
- TTL 越短，数据越实时，但 API 调用越频繁
- TTL 越长，API 调用越少，但数据可能不够实时

### 限流配置

| 配置项 | 说明 | 默认值 | 推荐范围 |
|--------|------|--------|----------|
| RATELIMIT_INTERVAL | 限流时间窗口（秒） | 60 | 30-300 |
| RATELIMIT_MAX_REQUESTS | 限流最大请求数 | 30 | 10-1000 |
| SEARCH_RATELIMIT_MAX | 搜索 API 限流 | 20 | 5-100 |

**调整建议**:
- **公共服务**: 使用默认值或更严格（20-30 次/分钟）
- **内部使用**: 可以放宽（100-200 次/分钟）
- **开发测试**: 设置较高限制（500-1000 次/分钟）
- **遭受攻击**: 临时降低限制（5-10 次/分钟）

**影响**:
- 限制越严格，越能防止滥用，但可能影响正常用户
- 限制越宽松，用户体验越好，但可能被滥用

## 配置优先级

系统按以下优先级读取配置：

```
1. 环境变量（.env 文件）
2. 数据库配置（后台管理设置）
3. 代码默认值
```

### 示例

如果同时存在：
- `.env` 文件: `CACHE_BALANCE_TTL=30`
- 数据库配置: `CACHE_BALANCE_TTL=60`
- 代码默认: `60`

最终使用：`30`（环境变量优先）

## 配置生效时间

### 立即生效

- API Keys 配置
- 限流配置
- 缓存 TTL 配置

### 需要清除缓存

某些配置修改后，建议清除相关缓存：

```bash
# 清除所有缓存
curl -X DELETE -H "Authorization: Bearer TOKEN" \
  "http://localhost:3000/api/admin/cache?type=all"

# 清除特定链的缓存
curl -X DELETE -H "Authorization: Bearer TOKEN" \
  "http://localhost:3000/api/admin/cache?type=prefix&pattern=eth"
```

## 使用场景

### 场景 1: API 调用受限

**问题**: 收到大量 429 错误（请求过多）

**解决方案**:
1. 增加缓存时间
   - `CACHE_BALANCE_TTL`: 60 → 300
   - `CACHE_TRANSACTIONS_TTL`: 120 → 600
2. 配置 API Keys（提高限制）
3. 清除现有缓存，让新配置生效

### 场景 2: 数据不够实时

**问题**: 用户反馈余额更新不及时

**解决方案**:
1. 减少缓存时间
   - `CACHE_BALANCE_TTL`: 60 → 30
   - `CACHE_TRANSACTIONS_TTL`: 120 → 60
2. 清除缓存
3. 监控 API 调用频率

### 场景 3: 遭受攻击

**问题**: 服务器负载过高，疑似被攻击

**解决方案**:
1. 立即降低限流阈值
   - `RATELIMIT_MAX_REQUESTS`: 30 → 10
   - `SEARCH_RATELIMIT_MAX`: 20 → 5
2. 查看黑名单，添加恶意 IP
3. 监控日志，分析攻击来源

### 场景 4: 开发测试

**问题**: 开发时频繁触发限流

**解决方案**:
1. 临时提高限流阈值
   - `RATELIMIT_MAX_REQUESTS`: 30 → 500
   - `SEARCH_RATELIMIT_MAX`: 20 → 100
2. 或者禁用限流（不推荐）
3. 测试完成后恢复默认值

## API 接口

### 获取配置

```http
GET /api/admin/settings
Authorization: Bearer <token>
```

**响应**:
```json
{
  "settings": [
    {
      "key": "CACHE_BALANCE_TTL",
      "value": "60",
      "description": "余额缓存时间（秒）"
    }
  ]
}
```

### 保存配置

```http
POST /api/admin/settings
Authorization: Bearer <token>
Content-Type: application/json

{
  "settings": [
    {
      "key": "CACHE_BALANCE_TTL",
      "value": "120",
      "description": "余额缓存时间（秒）"
    }
  ]
}
```

## 配置文件

### 环境变量 (.env)

```env
# API Keys
ETHERSCAN_API_KEY=your_key
TRONGRID_API_KEY=your_key
COINGECKO_API_KEY=your_key

# 缓存配置
CACHE_BALANCE_TTL=60
CACHE_TRANSACTIONS_TTL=120

# 限流配置
RATELIMIT_INTERVAL=60
RATELIMIT_MAX_REQUESTS=30
SEARCH_RATELIMIT_MAX=20
```

### 数据库配置

配置存储在 MongoDB 的 `settings` 集合中：

```javascript
{
  key: "CACHE_BALANCE_TTL",
  value: "60",
  description: "余额缓存时间（秒）",
  updatedAt: ISODate("2024-03-01T10:30:00Z")
}
```

## 监控和验证

### 验证配置是否生效

```bash
# 查看系统健康状态
curl http://localhost:3000/api/health

# 查看统计数据（包含配置信息）
curl -H "Authorization: Bearer TOKEN" \
  http://localhost:3000/api/admin/stats

# 测试限流
for i in {1..35}; do
  curl http://localhost:3000/api/search
  echo "Request $i"
done
```

### 查看日志

```bash
# 查看应用日志
pm2 logs crypto-wallet

# 查看限流日志
pm2 logs crypto-wallet | grep "ratelimit"

# 查看缓存日志
pm2 logs crypto-wallet | grep "cache"
```

## 最佳实践

### 1. 定期备份配置

```bash
# 导出配置
curl -H "Authorization: Bearer TOKEN" \
  "http://localhost:3000/api/admin/export?format=json" \
  > config-backup-$(date +%Y%m%d).json
```

### 2. 配置变更记录

建议记录每次配置变更：
- 变更时间
- 变更内容
- 变更原因
- 变更人员

### 3. 分环境配置

```env
# 开发环境
CACHE_BALANCE_TTL=10
RATELIMIT_MAX_REQUESTS=1000

# 测试环境
CACHE_BALANCE_TTL=30
RATELIMIT_MAX_REQUESTS=100

# 生产环境
CACHE_BALANCE_TTL=60
RATELIMIT_MAX_REQUESTS=30
```

### 4. 监控告警

设置告警阈值：
- 限流触发次数过多
- API 错误率过高
- 缓存命中率过低

## 安全建议

1. **保护管理员账号**
   - 使用强密码
   - 定期更换密码
   - 限制登录 IP

2. **审计日志**
   - 记录所有配置变更
   - 定期审查日志
   - 发现异常及时处理

3. **权限控制**
   - 只有管理员可以修改配置
   - 使用 Token 认证
   - Token 定期更换

4. **配置验证**
   - 输入范围验证
   - 防止恶意配置
   - 提供默认值保护

## 故障排查

### 问题 1: 配置不生效

**原因**:
- 环境变量优先级更高
- 缓存未清除
- 配置格式错误

**解决**:
1. 检查 `.env` 文件
2. 清除缓存
3. 重启应用

### 问题 2: 无法保存配置

**原因**:
- Token 过期
- MongoDB 连接失败
- 权限不足

**解决**:
1. 重新登录
2. 检查数据库连接
3. 查看错误日志

### 问题 3: 配置丢失

**原因**:
- 数据库数据丢失
- 配置被覆盖
- 缓存问题

**解决**:
1. 从备份恢复
2. 重新配置
3. 检查数据库
